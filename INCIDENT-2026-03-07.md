## 根本原因分析

### 问题代码

```javascript
// ❌ 错误：server/routes/backup.js 第 14-17 行
const BACKUP_DIR = path.join(process.env.HOME || '', '.openclaw', 'backups');

// 确保备份目录存在
if (!fs.existsSync(BACKUP_DIR)) {
  fs.mkdirSync(BACKUP_DIR, { recursive: true });
}
```

### 为什么会导致问题？

1. **模块导入时立即执行**：这段代码在文件顶层，Node.js 导入模块时立即执行
2. **同步阻塞 I/O**：`fs.existsSync()` 和 `fs.mkdirSync()` 是同步操作，阻塞事件循环
3. **服务器启动时执行**：每次 PM2 重启或热重载都会重新导入模块
4. **文件系统慢时恶化**：macOS 文件系统响应慢时，阻塞时间更长

### 为什么 PM2 重启 92 次？

1. 第一次启动：模块导入时执行同步 I/O，阻塞导致健康检查超时
2. PM2 检测到无响应：自动重启
3. 重启后再次导入模块：同样的阻塞再次发生
4. 循环往复：直到修复代码

```
┌─────────────────────────────────────────────────────────────┐
│  模块导入 → 同步 I/O 阻塞 → 健康检查超时 → PM2 重启        │
│      ↑                                                      │
│      └────────────── 循环 92 次 ─────────────────────────────┘
```

## 修复方案

### 代码修复

```javascript
// ✅ 正确：延迟初始化
const BACKUP_DIR = path.join(process.env.HOME || '', '.openclaw', 'backups');

// 封装为函数，按需调用
function ensureBackupDir() {
  if (!fs.existsSync(BACKUP_DIR)) {
    fs.mkdirSync(BACKUP_DIR, { recursive: true });
  }
}

// 在路由处理函数中调用
router.get('/list', (req, res) => {
  ensureBackupDir(); // 仅在需要时检查
  // ...
});
```

## 经验教训

### 🚫 禁止在模块顶层执行的操作

```javascript
// ❌ 禁止：同步文件 I/O
if (!fs.existsSync(path)) { fs.mkdirSync(path); }
const data = fs.readFileSync(file);

// ❌ 禁止：同步数据库操作
const db = sqlite3.openSync('db.sqlite');
const result = db.prepare('SELECT...').all();

// ❌ 禁止：同步网络请求
const response = http.get('http://...'); // 同步版本

// ❌ 禁止：耗时计算
const result = heavyComputation(); // 超过 100ms 的计算
```

### ✅ 推荐做法

```javascript
// ✅ 延迟到函数内部
function getData() {
  const data = fs.readFileSync(file);
  return data;
}

// ✅ 使用异步版本
const data = await fs.promises.readFile(file);

// ✅ 懒加载（首次使用时初始化）
let _instance = null;
function getInstance() {
  if (!_instance) {
    _instance = createExpensiveObject();
  }
  return _instance;
}

// ✅ 在应用启动时显式初始化（而不是模块导入时）
app.on('start', async () => {
  await ensureBackupDir();
  await connectDatabase();
});
```

### 📋 代码审查清单

添加新模块时检查：

- [ ] 是否有顶层同步 I/O 操作？
- [ ] 是否有顶层网络请求？
- [ ] 是否有顶层耗时计算（>100ms）？
- [ ] 所有初始化是否可以延迟到首次使用？
- [ ] 是否使用了异步版本（`fs.promises` vs `fs`）？
- [ ] 是否有超时和错误处理？

## 预防措施

### 1. 代码规范更新

在 `CONTRIBUTING.md` 中添加：

```markdown
## 模块编写规范

### 禁止顶层副作用

模块顶层（函数外部）禁止：
- 同步文件系统操作
- 同步数据库操作
- 同步网络请求
- 耗时计算（>100ms）

所有初始化应延迟到：
- 路由处理函数内部
- 应用启动钩子
- 首次使用时（懒加载）
```

### 2. 自动化检测脚本 ✅ 已实现

创建 `scripts/check-sync-io.js`，在 CI 和本地开发时运行：


### 3. ESLint 规则

添加规则检测顶层同步 I/O：

```javascript
// .eslintrc.js
rules: {
  'no-sync': 'error', // 禁止 sync 方法
  'no-process-exit': 'error',
}
```

### 5. 启动时健康检查

在 `server/index.js` 添加启动自检：

```javascript
// 启动时检查所有路由
async function healthCheck() {
  const routes = ['/api/status', '/api/backup/list', '/api/chat/messages'];
  for (const route of routes) {
    try {
      await fetch(`http://localhost:${PORT}${route}`);
    } catch (e) {
      console.error(`健康检查失败：${route}`, e);
      process.exit(1);
    }
  }
}

server.listen(PORT, () => {
  setTimeout(healthCheck, 1000); // 1 秒后检查
});
```
